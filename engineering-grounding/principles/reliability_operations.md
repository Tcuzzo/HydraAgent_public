# Reliability & Operations — Grounding Principles

Verified guardrails for changes that run live, persist data, or evolve an interface others depend on. Load-bearing because BACKS auto-commits to main and touches Postgres.

- **On an unexpected error, fail fast and loud — propagate or crash; never `except: pass`, never fabricate a success/empty return, never degrade silently.** A swallowed error is a latent fault that resurfaces as data corruption; prefer crash-only / let-it-crash with supervised restart over limping in an unknown state. Why: Gray, *Why Do Computers Stop and What Can Be Done About It* (fail-fast); Candea & Fox, *Crash-Only Software*; Erlang/OTP let-it-crash. Guards against the silent-swallow — the exact bug the operator's "no silent gates or fallbacks" invariant forbids.
- **Treat every schema change as an evolution problem: expand/contract, atomically reversible, and VERIFY AGAINST THE LIVE ARTIFACT (query the actual column/index/row), never the migration tracking table.** Add-nullable → dual-write → backfill → switch reads → drop; never drop/rename a column or table in one step, and never let an autonomous change touch data non-reversibly. A tracker can record APPLIED while doing nothing. Why: Kleppmann, *Designing Data-Intensive Applications* (backward/forward compatibility, rolling upgrades); the expand-contract / parallel-change pattern. Guards against the false-APPLIED migration class and one-step destructive DDL.
- **Instrument every autonomous action and service path with a structured, high-cardinality event (inputs, decision, outcome, IDs, timing) so a failure can be diagnosed by asking NEW questions of telemetry after the fact.** You cannot debug what you never instrumented; pre-built dashboards only answer questions you already knew to ask. Why: Google SRE Book (logs/metrics/traces); Majors/Fong-Jones/Miranda, *Observability Engineering* (unknown-unknowns, wide events). Guards against opaque runs where a failure leaves no trail — the gap that starves provenance and live-watch.
- **With enough users, every observable behavior of an interface becomes a de-facto contract — change it behind a version, deprecate on a clock, and never silently alter output shape/semantics a customer or downstream agent may parse.** Why: Hyrum's Law (Winters/Manshreck/Wright, *Software Engineering at Google*); Semantic Versioning. Guards against silent breaking changes to a live interface paying non-engineer customers or other agents depend on.
- **Land the smallest coherent atomic change — each independently revertable and `git bisect`-able — so a regression is located and rolled back without collateral.** Small batch size empirically correlates with BOTH higher throughput and lower change-failure rate. Why: Forsgren, Humble & Kim, *Accelerate* / DORA research. Guards against the big-bang commit that can't be cleanly bisected or reverted — acute because BACKS auto-commits to main.
