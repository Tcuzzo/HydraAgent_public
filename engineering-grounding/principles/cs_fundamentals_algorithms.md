# CS Fundamentals & Algorithms — Grounding for Build-Time

Verified, sourced guardrails for correctness, performance, and concurrency. Reason to these before you build; measure and prove, never assume.

- **Ship the clear correct version first; restructure for speed only after a profiler on realistic input names the hot path — then optimize that measured section.** ~97% of code is non-critical, so effort spent guessing costs clarity for zero throughput; the real wins hide in a ~3% only measurement finds (Knuth, *Structured Programming with go to Statements*, CACM Surveys 1974). Guards against micro-optimizing cold code from intuition AND the mirror abuse — quoting "premature optimization is the root of all evil" to dodge profiling a genuinely hot path.
- **Pick the structure/algorithm for the actual n, access pattern, and cache behavior — not the best asymptotic class; validate by measurement.** Big-O deliberately drops constants and low-order terms (CLRS Ch.3), so an O(n) contiguous scan crushes an O(log n) pointer-chasing structure until n is large — `std::vector` beats `std::list` even for mid-insert workloads because linked nodes scatter and every hop is a cache miss (Stroustrup, "Are lists evil?", isocpp 2014). Guards against reaching for the asymptotically-fanciest structure (tree/hashmap/linked list) on tiny/medium n where a flat array wins — "lower Big-O = faster" ignoring constants and cache lines.
- **Treat a green suite as evidence, never proof: state the invariant/postcondition first, then attack it with boundary, adversarial, and property-based inputs.** Testing exercises finitely many of a near-infinite behavior space, so it exhibits bugs but cannot certify their absence (Dijkstra, EWD249 §3, 1970); correctness comes from reasoning about invariants. Guards against "tests pass, therefore done" — especially a test that mocks the exact seam under change, which certifies nothing about live behavior.
- **Establish concurrency correctness by construction: one owner or one governing lock per shared mutable state, one global lock-acquisition order, and an argument over the interleavings.** Non-determinism means a race/deadlock can surface in 1 interleaving of millions, so 1000 clean runs prove nothing (Dijkstra's presence-not-absence at full force); reason over the happens-before partial order (Lamport, CACM 1978), and a consistent lock order structurally removes the circular-wait deadlock condition. Guards against "ran fine under load 1000×, so it's thread-safe" — shipping shared-state concurrency validated only by repeated execution.
- **Before adding threads/workers/fanned-out agents, estimate the serial fraction and expect coordination cost to grow — past some point more workers make throughput go *retrograde*; find that turn-around by measuring, never by computing an "optimum."** Amdahl (AFIPS 1967) caps speedup at 1/serial-fraction (5% serial ⇒ never beyond 20×) — that part is exact. Gunther's USL shows contention + coherency costs eventually drive throughput retrograde, but its βN² is an empirical fit to *cache-coherent processors*, NOT to LLM-agent fan-out (agents share no cache-coherent state), so take only the qualitative law — "more coordination eventually loses" — and locate the real peak empirically. Guards against "just add more workers, it scales linearly" AND against false precision: dressing agent fan-out in a hardware coherency coefficient no one has fit, borrowing a rigor it hasn't earned (the exact SE-folklore move this grounding exists to reject).
- **In CPython, don't use threads for CPU-bound work — reach for multiprocessing, GIL-releasing native extensions, or a free-threaded build; keep threads/asyncio for I/O waiting.** The GIL lets only one thread run Python bytecode at a time, so a CPU-bound ThreadPool runs effectively serially and switch overhead can make it slower than single-threaded; I/O threads help because blocking calls release the GIL (CPython C-API threads docs; PEP 703 motivation). Guards against wrapping a CPU-bound loop in a ThreadPool and expecting multi-core speedup.
- **Never compare floats with `==` (use an explicit tolerance), and never store or compute money in binary float — use integer minor units (cents) or an exact decimal.** IEEE-754 can't represent most decimal fractions exactly (0.1 + 0.2 ≠ 0.3), rounding error accumulates, and addition is non-associative so summation order changes the result (Goldberg, CACM Surveys 1991; IEEE 754). Guards against `if price == 0.30:`, float running-totals, and float balances — a correctness/audit defect, not a rounding nicety.
- **Explicitly handle and test the boundaries — empty, single element, min/max — and compute midpoints as `lo + (hi - lo) / 2`, never `(lo + hi) / 2`.** A textbook-correct binary search carried a `(low+high)/2` integer-overflow bug latent ~9 years in the JDK, tripping once the array exceeds ~2³⁰ (Bloch, "Nearly All Binary Searches and Mergesorts are Broken", 2006); off-by-one/empty/overflow are the highest-density defect region because they sit outside the inputs a developer pictures. Guards against "works on the example input" — skipping the empty/singleton/max/overflow cases.
