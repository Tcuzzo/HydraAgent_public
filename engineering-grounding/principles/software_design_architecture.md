# Design & Architecture — Grounding Principles

Verified guardrails from the foundational software-architecture literature. Apply before drawing a module boundary, adding a layer or abstraction, or wiring an integration point.

- **Draw module boundaries around what will change, not around processing steps.** Each module hides one volatile decision (data format, schema, provider, wire protocol, algorithm) behind a stable interface — name that decision before creating the boundary; if it hides nothing changeable, the boundary is wrong. Why: Parnas, *On the Criteria To Be Used in Decomposing Systems into Modules* (CACM 1972) — information-hiding decomposition lets modules be built, understood, and changed independently. Guards against functional/flowchart decomposition, where one changeable assumption is smeared across every module and one change ripples through all of them.
- **Classify complexity as essential vs. accidental before adopting any framework, pattern, DSL, agent, or new model.** Claim leverage only on accidental cost (boilerplate, glue, formatting); reject any "solution" that adds indirection or layers while leaving the domain's essential complexity untouched. Why: Brooks, *No Silver Bullet* (IEEE Computer 1987) — accidental cost is already small, so no single technique yields an order-of-magnitude gain on the whole. Guards against silver-bullet over-tooling that stacks new accidental complexity (meta-frameworks, extra layers, an LLM) on unchanged essential complexity.
- **Give every call crossing a process, host, or network boundary an interface that surfaces latency, partial failure, and concurrency.** Never reuse a plain in-process signature for a remote call; design assuming the network is unreliable, non-zero-latency, finite-bandwidth, and insecure. Why: Waldo et al., *A Note on Distributed Computing* (Sun Labs TR-94-29, 1994) + the eight fallacies of distributed computing — the local/remote distinction is irreducible; a caller blind to partial failure cannot handle it. Guards against distribution transparency (modeling an RPC / HTTP / DB / model-API call as if it were local).
- **Wrap every outbound integration point in a timeout, a bounded backoff-with-jitter retry, and a circuit breaker; isolate resource pools with bulkheads.** No timeout-less or unbounded-retry outbound call ships. Why: Nygard, *Release It!* (Stability Patterns) — integration points are the #1 production killer; one blocked or slow dependency drains a shared pool and cascades into total outage. Guards against timeout-less / unbounded-retry calls on shared pools that let one degraded dependency take down the whole system.
- **Make every state-changing handler idempotent; treat delivery as at-least-once.** Use natural keys, dedup tokens, conditional/compare-and-set writes, or upserts so repeated delivery has the same effect as one — never rely on exactly-once. Why: Helland, *Idempotence Is Not a Medical Condition* (CACM 2012) — reliable systems must retry, so idempotence is the property that makes retries safe. Guards against exactly-once assumptions in non-idempotent handlers that double-charge, double-insert, or double-apply on redelivery.
- **Build the minimum abstraction the present, concrete requirement needs; generalize only on the second or third real use.** No config knobs, plugin seams, extra layers, or patterns for imagined futures. Why: Fowler, *YAGNI* (bliki) and *Refactoring* (Speculative Generality smell) — speculative features carry opportunity cost, cost of carry, and misalignment (the guessed shape rarely fits the real future need). Guards against premature abstraction and patternitis — extension points and framework layers for futures that never arrive or don't fit.
- **Make each component do one thing well behind the simplest interface a real, present client needs; split the normal case and the worst case onto separate code paths.** Don't generalize past a demonstrated need — when in doubt, leave it out. Why: Lampson, *Hints for Computer System Design* (SOSP 1983) — trading breadth for correctness and speed is what survives production. Guards against over-generalized do-everything interfaces and a single path straining to cover both common and exceptional cases, which hides the fast normal path and multiplies failure modes.
- **Put every correctness-critical check (validation, retry, dedup, idempotency) at the endpoint that holds the true invariant; treat any lower-layer version as a performance optimization only.** The application still validates even when a layer below claims it already did. Why: Saltzer, Reed & Clark, *End-to-End Arguments in System Design* (ACM TOCS 1984) — a function is complete and correct only with the endpoints' knowledge of the real correctness condition. Guards against trusting transport "reliability," framework sanitization, or a queue's delivery guarantee as if it were the end-to-end guarantee.
