# Performance & Systems Engineering — Grounding Principles

Intent: before you optimize, prove where the time goes and that the win is worth building; measure with rigor, and design against the tail — not the average.

- **Profile before you optimize; ship correct-and-simple first.** Knuth's "critical 3%" is found by measurement, not intuition — the conditional half of his 1974 rule (ACM CS 6(4)) that the cargo-culted version drops; the USE method (Gregg) systematizes the read. Guards against the "this loop looks slow, let me hand-tune it" reflex that lands effort on code that never dominated runtime.
- **Refuse any build whose target is not the dominant wall-clock fraction.** Amdahl's Law (AFIPS 1967) caps overall speedup at 1/((1−p)+p/s) and at 1/(1−p) even as s→∞, so a 10× win on a 5%-of-runtime stage buys ~1.045× overall — turn "is this worth building?" into an arithmetic pre-check. Guards against pouring a GPU port / Rust rewrite / cache layer into a minor slice and being shocked p50 barely moves.
- **Classify the bottleneck layer (CPU / memory / disk / network) before choosing the fix.** The Dean-Norvig latency hierarchy spans ~9 orders of magnitude (L1 ~1ns → RAM ~100ns → SSD read ~150µs → DC RTT ~500µs → cross-continent ~150ms), so a faster-language rewrite is a constant-factor win that cannot recover time spent in a round trip or a seek; USE (Gregg) locates the layer without familiarity bias. Guards against "rewrite it in Rust/C++" or micro-optimizing CPU code while the process is I/O- or network-bound.
- **Cross an FFI/native boundary only when per-call work swamps the crossing cost — batch and keep data native.** The fixed per-call tax (GIL acquire + type conversion) means a compiled path only pays off past a large-enough batch; a controlled toolchain study (arXiv:2507.00264) found the relationship inverts only around batch size ~10, and NumPy/SciPy win because one vectorized call buries the overhead (pythonspeed.com). Measure the crossover; it is workload-dependent. Guards against chatty scalar FFI that hands the entire win back to boundary + marshalling overhead.
- **Set and gate latency SLOs on the tail (p99, p99.9, max), never the mean or median.** Under fan-out to N backends the odds a request hits at least one slow node approach 1, so the tail becomes the typical experience, and averages are engineered to hide it — Dean & Barroso, "The Tail at Scale" (CACM 56(2), 2013), which also prescribes the real fix (hedged/tied requests, prioritization). Guards against "average latency looks fine" while a single slow percentile dominates every fan-out request.
- **Benchmark with warmup and many iterations; report an interval, and always measure one level deeper.** Non-determinism makes single-run conclusions unsound — treat a difference as real only when confidence intervals don't overlap (Georges et al., OOPSLA 2007) — and decompose the top-line number into its components or it's "more marketing than science" (Ousterhout, CACM 61(7), 2018). Guards against "it's 20% faster" from one un-warmed run with no variance and no account of why it changed.
- **Load-test open-loop at a constant target rate and correct for coordinated omission (or use wrk2 / HdrHistogram).** A closed-loop harness that blocks in lockstep with the system under test stops issuing requests during a stall, omitting exactly the slow samples that define the tail and undercounting p99 by orders of magnitude (Tene, "How NOT to Measure Latency"; HdrHistogram re-synthesizes the missing samples). Guards against trusting a stock load generator that "proves" a p99 production will never see.
- **Size pools and queues with Little's Law; raise throughput by cutting service time or adding capacity, not concurrency.** L = λW is an exact steady-state identity (Little, Operations Research 9(3), 1961), so past a saturated resource service time is fixed and added concurrency only grows queue depth and tail latency (Gregg, *Systems Performance* — distinguishing "busy" from "drowning"). Guards against "add more threads/workers to make it faster" against an already-saturated resource, which lengthens the queue and adds no throughput.
- **Bound cost-per-unit-of-work the way you bound latency: right-size the model to the job, exploit prompt/prefix caching and batching, and treat a cheaper harness that clears the quality bar as strictly superior.** Never spend frontier tokens where a cached or local path is PROVEN equivalent — dollars/tokens are an engineering axis, not an afterthought. Why: Amdahl-by-analogy (invest where the dominant COST fraction lives) + provider prompt-caching economics. Guards against reflexively reaching for the biggest model, re-sending uncached context, and ignoring cost until the bill arrives — directly serves "dumb model + smart harness = elite" for the priced-out.
